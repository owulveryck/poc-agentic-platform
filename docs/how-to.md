# How-to guides — targeted recipes

> Each guide solves **one** concrete problem. They assume you have completed
> the [tutorial](tutorial.md).

## 1 — Add a new architectural invariant (ADR)

1. Create `adr/ADR-0XX-my-invariant.md` with the YAML front matter described
   in [reference.md](reference.md#adr-front-matter).
2. Apply the **classification test**: *"more useful, or useless, when the
   model is twice as intelligent?"*
   - More useful → `nature: amplifier`, `sunset_condition: null`.
   - Useless → `nature: compensatory` + a **mandatory, measurable**
     `sunset_condition`.
3. Fill `scope_selectors` with the keywords that make the invariant relevant.
4. State the invariant **semantically**: never "modify file X at line Y".
5. Verify: `curl -X POST /enrich` with an intent containing a selector; the
   invariant must be returned.

## 2 — Add a deterministic validation policy

1. Implement the check in `internal/linter/linter.go` (`Validate`).
2. Register the policy in `Registry` with its `Nature` and, if compensatory,
   its `SunsetCondition`.
3. If the policy materializes an ADR, set `enforcement.policy_id` in the ADR
   (keep both consistent).
4. Add a test in `internal/linter/linter_test.go`.
5. Run `go test ./internal/debt/` to confirm the debt ratio stays acceptable.

## 3 — Retire compensatory scaffolding (pay back the debt)

1. Verify the `sunset_condition` is met (new model benchmark).
2. Remove the check and its `Registry` entry (or the translator).
3. If the underlying intent remains valid, **promote** it to an amplifier
   invariant instead of deleting it.
4. Mark the source ADR `status: superseded` and document the retirement.
5. Re-run the test suite: the transition-debt ratio must have decreased.

## 4 — Add a Smart Platform Tool

1. Create a package under `internal/smarttools/<name>/` implementing the
   `smarttools.Tool` interface (`ID`, `Run`).
2. Sandbox first: act on an isolated copy, never directly on the target.
3. In the analysis step, call `translate.Generic` for the raw outcome
   (compensatory) then a **semantic enricher** with the business context the
   model cannot guess (amplifier). Aim for an *actionable*
   `remediation_guidance`, not a descriptive one.
4. Register it in `cmd/ppg/main.go` with `smarttools.Register`.
5. The ticket guard is inherited: any out-of-scope call is refused before
   your code runs.

## 5 — Connect a black-box agent (GitHub Copilot / Cursor)

1. Start the gateway, then run the pre-flight with your intent:

   ```bash
   go run ./adapters/preflight "Add the Seka payment method to checkout"
   ```

2. Check the generated `.github/copilot-instructions.md` (and `.cursorrules`).
   Copilot reads it natively as repository custom instructions:

   ```markdown
   # Platform context (generated by the Platform Planning Gateway)

   Intent: Add the Seka payment method to checkout

   ## ADR-042

   ## Invariant
   Every outbound call to a third-party service (payment, KYC, notification)
   MUST go through the corporate security egress proxy (`security-egress-proxy`).
   ...
   ```

3. From there, Copilot's black-box planning is steered by the same invariants
   the gateway serves to every other agent.
4. ⚠ Limit: hard gating is **not** guaranteed with black boxes: no hook can
   intercept their edits. The locked-plan check must then happen at apply
   time (pre-push platform CLI) instead of in-loop.

## 6 — Connect Claude Code (MCP + hooks)

Full walkthrough in [adapters/claudecode/README.md](../adapters/claudecode/README.md).
The short version:

1. **Planning (pillar 1)**: register the MCP server in the target project:

   ```bash
   claude mcp add ppg -- go run /path/to/poc-agentic-platform/adapters/claudecode/mcpserver
   ```

   Claude Code now sees `get_platform_guidelines_for_intent` and
   `lock_in_plan` as native tools. A successful lock writes the capability
   ticket to `.ppg-ticket`.

2. **Gating (pillar 2)**: build the guard and register it as a `PreToolUse`
   hook on `Edit|Write` in `.claude/settings.json`:

   ```bash
   go build -o ~/.local/bin/ppg-guard ./adapters/claudecode/guard
   ```

   ```json
   { "hooks": { "PreToolUse": [ { "matcher": "Edit|Write",
       "hooks": [ { "type": "command", "command": "ppg-guard", "args": [] } ] } ] } }
   ```

3. **Contract**: copy [adapters/claudecode/CLAUDE.example.md](../adapters/claudecode/CLAUDE.example.md)
   into the target project's `CLAUDE.md`.

4. **Verify**: ask Claude to edit a file outside the locked plan: the hook
   exits with code 2 *before* the edit executes, and the model receives the
   semantic refusal (`OUT_OF_PLAN_SCOPE: ... re-plan through lock_in_plan`),
   which steers it back to the paved road on its own.
